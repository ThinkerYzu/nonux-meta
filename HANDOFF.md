6. **Vector I/O syscalls beyond writev/readv (opportunistic, not blocking).**  Slice 7.6c.3c added `NX_SYS_WRITEV` and slice 7.6d.N.6a added `NX_SYS_READV` because musl's `__stdio_write` and `__stdio_read` require them.  The natural follow-on family stays unimplemented — programs that hit any of these get `-ENOSYS` from the syscall-translation layer:
   - **`pwritev` / `preadv`** (`__NR_pwritev = 70` / `__NR_preadv = 69`) — positional vector I/O (read/write at an offset without modifying the seek cursor).  Pulled in by programs that want lock-free seeking from multiple threads to one fd; busybox doesn't use them.  Add when a real demo forces it.
   - **`pwritev2` / `preadv2`** (`__NR_pwritev2 = 287` / `__NR_preadv2 = 286`) — Linux-specific extensions with RWF_* flags (O_DSYNC, O_APPEND, RWF_HIPRI).  Almost no userspace uses them; defer indefinitely.
   - **`process_vm_writev` / `process_vm_readv`** (`__NR_process_vm_writev = 271` / `__NR_process_vm_readv = 270`) — cross-process memory access.  Only useful with multi-process debuggers/tracers (gdb, strace).  Big departure from our current "user-window-per-process" memory model — defer until a real cross-process tool lands in the project.

   Implementation pattern is already established by sys_writev: copy_from_user the iovec array (cap 16 entries), dispatch each entry, stop on first short transfer per Linux convention, return running total.  Each addition is ~30 lines of kernel + 1 line each in syscall_arch.h's translation table and syscall_cp.s's asm switch.

7. **Tolerable-syscall stubs sweep — split between slices 7.6d.N.12 (sigaction family) and 7.6d.N.13 (everything else).**  Slice 7.6d.N.4's diagnostic instrumentation (temporary musl pass-through + kernel `kprintf("[svc?=N]")`, both reverted before commit) captured 11 unmapped Linux syscalls that ash's startup hits.  Two of them — `rt_sigaction` (134) and `rt_sigprocmask` (135) — need real handler dispatch infrastructure for the interactive Ctrl-C path at N.final, so they're scheduled into N.12 with the dispatch trampoline + per-process `sigaction[NSIG]` table.  The remaining 9 are pure stubs scheduled into N.13:
   - **`set_tid_address`** (`__NR_set_tid_address = 96`) — return a fake tid (e.g., 1).  musl's `__post_Fork` calls this in the child to register robust-mutex cleanup; we don't have robust mutexes, so the value is don't-care.
   - **`getuid` / `geteuid` / `getgid` / `getegid`** (`__NR_getuid = 174`, `geteuid = 175`, `getgid = 176`, `getegid = 177`) — return 0 (single-user kernel, root by default).  ash and `id` both consult these; returning 0 lets `id` print `uid=0(root) gid=0(root)`.
   - **`setuid` / `setgid`** (`__NR_setuid = 146`, `setgid = 144`) — return 0 success.  Single-user kernel, no actual privilege boundary; ash calls these defensively.
   - **`getpid` / `getppid`** (`__NR_getpid = 172`, `getppid = 173`) — return current `nx_process->pid` / `nx_process->parent_pid` (the latter wired in slice 7.6d.N.6b).  `ps`/`pgrep` need real values eventually.
   - **`uname`** (`__NR_uname = 160`) — fill `struct utsname` with `sysname="nonux"`, `nodename="nonux"`, `release="v0"`, `version="aarch64-qemu-virt"`, `machine="aarch64"`.  ash and many shell scripts consult this.

   Slice 7.6d.N.13 is mostly one big `framework/syscall.c` block + 9 entries each in `syscall_arch.h` and `syscall_cp.s`.  Adds 9 new `NX_SYS_*` numbers (or one shared dispatcher number with a sub-opcode if we want to keep the table small).

8. **Handle encoding's generation-in-high-bits collides with POSIX-fd arithmetic — scheduled into slice 7.6d.N.16.**  Slice 7.6d.N.11 uncovered: ash's `xdup_CLOEXEC_above(fd, avoid_fd)` issues `fcntl(fd, F_DUPFD_CLOEXEC, avoid_fd + 1)`; with our encoding `(generation << 8) | (idx + 1)`, an avoid_fd whose slot has been closed-and-reopened encodes as ≥ 257 even though the table only has 64 slots — so `arg + 1` overflows the cap and `sys_fcntl` returns `EINVAL`.  Patched in 7.6d.N.11 by falling back to `min_idx = 0` when `arg ≥ NX_HANDLE_TABLE_CAPACITY`; preserves ash's intent (loop skips non-INVALID slots) but strictly violates POSIX F_DUPFD's "lowest fd ≥ arg" guarantee.  N.16 drops the in-handle generation entirely and relies on per-slot in-table generation strictly for stale-handle detection — encoded value becomes `idx + 1`, matching POSIX `lowest fd` semantics and Linux's "reuse small fd numbers across close+open" behaviour.  Slice 5.3's stale-handle invariants restated against the new encoding (the test that handle-after-close fails to lookup still holds, but via the in-table generation comparison rather than encoded-value mismatch).  Skippable if N.final's path doesn't surface a second arithmetic-overflow (the N.11 fcntl band-aid covers the only case currently observed).

Each slice-7.4 / 7.5 / 7.6a / 7.6d ktest today leaks the full child stack: `sched_rr_purge_user_tasks` only `nx_list_remove`s tasks from the runqueue — it doesn't free the `nx_task` struct, the kstack, the owning `nx_process`, the process's MMU address space, or the 8 MiB user-window backing.    - Per-test quantum override.  `SCHED_RR_DEFAULT_QUANTUM_TICKS = 2` is a compromise.  A `kernel.json` knob (once gen-config emits per-component config macros) lets tests tune per-build.
   - C-compiled EL0 crt0 for `main(argc, argv)` entry — needed when busybox expects POSIX main semantics.
2. **Phase 5 follow-ups (deferred across 5.5 / 5.6):**

   - `nx_process_exit(code)` + scheduler TASK_ZOMBIE handling so EL0 tasks can retire cleanly instead of stranding in a `wfe` loop.

   - `copy_from_user` with page-fault fixup — the current version bounds-checks but doesn't handle faults gracefully.

8. **Real per-process memory management (long-term, post-Phase-7).**  The current "every process gets one contiguous 8 MiB physical block, mapped via four 2 MiB L2 blocks" model is functional but wasteful and rigid:
   - **Over-commit:** every process eats 8 MiB of physical RAM regardless of actual use.  `init_prog` is ~800 bytes; signal demo ~250 bytes.  Most of the per-process backing is never touched.
   - **PMM fragmentation:** each process_create needs 10 MiB physically contiguous (8 MiB usable + 2 MiB alignment slack).  After many fork/exec cycles, PMM may not find a contiguous 10 MiB run even with hundreds of MiB free elsewhere.  Eventually a long-running system can't fork.
   - **Fork is O(USER_WINDOW_SIZE) memcpy** even when the child overwrites two bytes and exits.  No COW.
   - **Hard 8 MiB ceiling per process.**  Any binary whose code+data+heap+stack would exceed ~6 MiB of mapped memory just can't run, and we can't grow the heap past `NX_PROCESS_HEAP_LIMIT` without resizing the entire window.
   - **Wasted alignment slack.**  Each malloc carries 2 MiB of slack that's allocated to the process but unused — locked away from PMM until the process is destroyed.

   The right direction is the standard one: **drop 2 MiB block descriptors for L3 4 KiB pages, give each region (text / data / heap / stack) its own VMA, and add demand paging + copy-on-write for fork.**  Concrete shape:

   - Per-process page tables grow a third level (L3 4 KiB pages instead of L2 2 MiB blocks).  4 KiB granularity makes per-page demand-paging and per-page COW natural.
   - `nx_process` gains a list of VMAs (`{vaddr_lo, vaddr_hi, prot, file_or_anon, offset}`).  ELF loader populates one VMA per PT_LOAD; brk grows a heap VMA up; mmap creates new VMAs.
   - Page-fault handler upgrades from "panic" to "look up VMA, allocate one PMM page, populate L3 entry, return."  COW pages get a "copy on write" flag; first write demotes the entry from RO to RW with a fresh page.
   - `mmu_copy_user_backing` becomes `mmu_clone_user_backing_cow`: walks parent's L3, marks each entry RO in both parent and child, no memcpy.  First write in either side faults, copies one page, splits.
   - `mmap(NULL, size, PROT_*, MAP_ANON|MAP_PRIVATE, ...)` falls out for free as a new VMA + lazy population.  busybox + most real userspace stops needing brk.

   This is a multi-slice rework — page table layout (4 KiB granule), VMA framework, fault handler, fork rework, mmap, and migration of existing tests off the contiguous-block assumption.  Phase 7 explicitly stays on the simple model so we can ship busybox; the rework lands in a dedicated phase, probably between Phase 8 (recomposition) and Phase 9 (benchmarks) or as part of Phase 9's perf-oriented pass.  Worth a one-line callout in DESIGN.md so future readers don't think the current scheme is the long-term plan.
