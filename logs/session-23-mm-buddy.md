## What Was Done

### `interfaces/mm.h` ‚ÄĒ new

Universal ops contract for `memory.page_alloc` implementations:

- `void *alloc_pages(self, order)` ‚ÄĒ returns `2^order` contiguous
  pages, page-aligned, or NULL.
- `void  free_pages(self, ptr, order)` ‚ÄĒ release a block. Order MUST
  match the matching alloc; the allocator is not required to remember
  per-block sizes.
- `size_t page_size(self)` ‚ÄĒ typically 4096.
- `unsigned max_order(self)` ‚ÄĒ largest supported order, inclusive.

Ownership is explicit in the header: the returned pointer is
caller-owned between alloc and free, and the allocator never touches
the backing storage while it's owned by the caller. Concurrency note
locks for SMP to the future.

### `components/mm_buddy/{manifest.json,mm_buddy.c,README.md}` ‚ÄĒ new

Classic power-of-two buddy on a fixed 16-page per-instance pool (64 KiB
at 4 KiB granule, `MM_BUDDY_POOL_ORDER = 4`):

- Pool allocated in `init()` via `posix_memalign` on host / `kheap`'s
  `malloc` on kernel ‚ÄĒ both return page-aligned pointers.
- Free blocks carry an intrusive `struct free_node` (one `next` pointer)
  at their head ‚ÄĒ classic intrusive free-list, no side-table.
- `alloc_pages(order)`: find smallest free block at order ‚Č• target,
  split down, return the low half.
- `free_pages(ptr, order)`: push onto `free_lists[order]`; if the buddy
  (address XOR block_size, relative to pool base) is also free, remove
  it and coalesce, recurse up to `MM_BUDDY_POOL_ORDER`.
- Lifecycle: `init` allocates pool + seeds the top-level free list;
  `enable/disable` are counter-only no-ops; `destroy` frees the pool.
  100√ó cycling tested end-to-end on host with zero residue.

### `test/host/conformance/conformance_mm.{h,c}` ‚ÄĒ new

Seven universal cases matching the interface contract:

1. `page_size_is_positive_power_of_two` ‚ÄĒ sanity.
2. `max_order_is_supported` ‚ÄĒ max_order > 0, alloc at max_order
   succeeds then frees cleanly.
3. `alloc_order_zero_is_page_aligned_and_nonnull`.
4. `alloc_order_zero_twice_returns_distinct_pages` ‚ÄĒ no aliasing,
   pointers at least page_size apart.
5. `alloc_beyond_max_order_returns_null`.
6. `max_order_alloc_uses_full_pool` ‚ÄĒ after a max-order alloc, even
   order-0 fails; after free, max-order alloc works again (proves
   coalescing).
7. `alloc_free_roundtrip_is_coalescing` ‚ÄĒ allocate every page at order
   0, free them all, then alloc at max_order must succeed.

## Key Decisions

- **Per-instance pool, not a global one.** Each bound mm_buddy
  component owns its own 64 KiB pool. Two instances never share
  state; the conformance fixture can `create()` and `destroy()`
  freely. When a slice 8 / 9 consumer needs a bigger shared pool, a
  sibling component (`mm_buddy_large` or similar) is one manifest +
  `#define` away.
- **64 KiB pool (max_order = 4) is the default.** Small enough for
  the host conformance suite to run instantly on every test, big
  enough to exercise splits and merges end-to-end. Near-term
  consumers (page-table pages in 5.5, handle tables in 5.3) need
  ‚Č§ 8 pages total.
- **Intrusive free-list via `struct free_node` at block head.**
  Zero side-table overhead, O(1) push/pop. Removal during coalescing
  is an O(pool) scan, which is fine at 16-entry pool size.
- **Page alignment only, not block alignment.** The interface
  explicitly says callers must not rely on alignment beyond
  `page_size`. Kernel's `kheap` returns page-aligned pointers for
  whole-page allocations; host uses `posix_memalign(page_size)`.
  Higher-order block alignment depends on the pool base's alignment
  mod block_size, which is not guaranteed ‚ÄĒ a deliberate narrowing
  of the contract so the allocator doesn't have to over-allocate and
  waste pages finding aligned sub-regions.
- **`iface_ops` in the descriptor, same wire-up as `sched_rr`.**
  The kernel test reads `s->active->descriptor->iface_ops` and casts
  to `const struct nx_mm_ops *`. Phase 5 has more consumers that
  need this (handle framework, MMU's page-table allocator), which
  will all follow the same "lookup slot ‚Üí active ‚Üí descriptor ‚Üí
  iface_ops" pattern.
- **Pool survives enable/disable, not destroy.** `disable()` keeps
  the pool carved ‚ÄĒ held allocations remain valid across a pause,
  so a recomposition that bounces the component doesn't corrupt
  downstream state. Only `destroy` releases the pool. Matches the
  sched_rr pattern where `disable` doesn't touch the runqueue.

## Known Issues / Follow-ups

- The 64 KiB pool is a compile-time constant. A runtime knob (via
  `kernel.json` `"config": { "pool_order": 6 }`) lands when gen-config
  starts emitting per-component config macros ‚ÄĒ not blocking 5.3.
- mm_buddy doesn't yet own the MMU's boot-time page tables. Slice 5.1
  still puts L1/L2 tables in BSS. Migration to mm_buddy-owned tables
  is a follow-up that pairs naturally with slice 5.5's per-process
  TTBR0 bring-up.
- `LOAD segment with RWX permissions` linker warning still fires.
  Splitting kernel segments into separate PHDRS with per-section
  PXN/AP is a slice 5.5-adjacent cleanup.
