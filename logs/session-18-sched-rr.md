## What Was Done

### `components/sched_rr/` — new component

- **`manifest.json`** — `name: sched_rr`, `version: 0.1.0`, `iface:
  scheduler`. No `requires`, no `spawns_threads`, no `pause_hook` —
  round-robin has no worker threads, and its runqueue is pure
  internal state, so `ops->pause` (when 4.4 wires it) just stops
  accepting new enqueues.
- **`sched_rr.c`** (~180 LOC).
  - `struct sched_rr_state`: `nx_list_head runqueue`, `current`,
    `quantum_ticks`, `remaining`, plus four observable counters
    (`init/enable/disable/destroy_called`).
  - `const struct nx_scheduler_ops sched_rr_scheduler_ops` — the
    interface surface the core driver's `sched_init` will consume in
    4.4. Enqueue → tail, pick_next → head (idempotent), dequeue scan,
    yield rotates head-to-tail and resets the quantum, tick
    decrements `remaining` and flips `need_resched` at zero,
    set_priority returns `NX_EINVAL` (round-robin doesn't honour
    priorities). Enqueue detects double-enqueue via a linear scan and
    returns `NX_EEXIST`.
  - `const struct nx_component_ops sched_rr_component_ops` — lifecycle
    verbs only. `handle_msg` intentionally NULL (the scheduler is
    called directly, not IPC-driven).
  - `NX_COMPONENT_REGISTER_NO_DEPS` emits the descriptor into the
    `nx_components` linker section; bootstrap walks it at boot.
- **`README.md`** — state-struct shape, symbol table, rationale for the
  component (interface → conformance → component path), forward
  pointers at 4.4 + 3.9b + Phase 8.

### Bug fix — `framework/bootstrap.c` silent slot drop

Caught while debugging the first failing kernel test: only the first
slot in `nx_boot_slots[]` was being registered at boot, even though
both entries were present in the linked binary and
`nx_boot_slots_count` at runtime was 2.

Root cause: the slice-3.9a weak fallback in `bootstrap.c` was
`__attribute__((weak)) const unsigned nx_boot_slots_count;` — a weak
tentative definition with `const`. Under `-O2`, GCC treats this as a
compile-time `0` inside the TU and constant-folds the bound out of
the slot-register loop, turning it into `for (i = 0; i < 0; i++)` —
dead code. The strong definition in `gen/slot_table.c` *did* win the
link (so `printf("%u", nx_boot_slots_count)` printed 2), but every
use of the symbol inside `bootstrap.c`'s control flow was folded
before the link happened.

Fix: dropped the weak fallbacks entirely. `gen/slot_table.c` is
always generated by the Makefile's cascade rules now (slice 3.9a),
so the fallbacks were only a theoretical safety net for unit-test
builds that don't compile `bootstrap.c` — and those don't exist.
`framework/bootstrap.h`'s `extern` declarations are now the sole
decl, forcing the compiler to emit a real runtime load on every
read.

## Decisions

**Two separate ops tables (component ops + scheduler ops).** The
manifest-facing `nx_component_ops` handles lifecycle and IPC
dispatch; `nx_scheduler_ops` is the interface-specific surface the
core driver calls directly. Exporting both as public `const` symbols
is the simplest possible shape. Slice 4.4 will teach `framework/
bootstrap.c` to find `<manifest>_scheduler_ops` by name convention
(or via an added `iface_ops` field on the descriptor) — for now the
symbol just needs to exist.

**No inline state struct in a header.** Followed the uart_pl011
precedent: state is private to `sched_rr.c`. Kernel tests mirror the
struct layout locally (same trick as `ktest_bootstrap.c` for
uart_pl011_state). Fragile if fields reorder, but fails loudly when
it does. Host tests deliberately don't peek at the counters —
lifecycle health is observable via return codes + mem_track; the
counter mirror is a kernel-test convenience.

**`handle_msg` stays NULL.** The scheduler is driven through
`scheduler_ops`, not IPC. If a future "send a message to the
scheduler" use case appears (slice 3.9b runtime introspection? ),
handle_msg can be added with a msg_type dispatch table then.

**`yield` resets the quantum.** When the current task voluntarily
yields, it's fair to give the next task a full quantum — otherwise
the residue of the yielding task's slice bleeds into its successor.
Matches Linux's `sched_yield` behaviour.

**`NX_HOOK_CONTEXT_SWITCH` enum + arm together.** Adding the enum
without the matching union arm would mean any hook handler that
switched on `point` would face an incomplete context. Enum + arm
are a single atomic addition.

**Dropped the weak fallback in `bootstrap.c`.** Debugging the
silent-slot-drop bug revealed the fallback was (a) almost never
actually used (no host test compiles `bootstrap.c`), and (b) the
source of the subtle `const`+weak constant-folding that broke slice
4.3's multi-slot bring-up. Simpler to make the extern decls
authoritative.
