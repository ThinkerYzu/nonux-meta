# Session 113: Idle loop runqueue size check

**Date:** 2026-05-06
**Phase:** Post-Phase-9b (between Phase 9b and Phase 9)
**Branch:** master

---

## Goals

- Make the idle task yield immediately when there are runnable tasks, rather than relying solely on the `need_resched` + IRQ-return path.

## What Was Done

Added a `runqueue_size` op to the `scheduler` interface so the idle loop can check whether non-idle tasks are waiting before deciding to sleep again after `wfi` returns.

### IDL change

Added `runqueue_size` (op_id 7) to `interfaces/idl/scheduler.json` with `returns: int_status` and no params.  Ran `make gen-iface` to regenerate:
- `interfaces/scheduler.h` — `int (*runqueue_size)(void *self)` in `struct nx_scheduler_ops`
- `interfaces/scheduler_msg.h` — `NX_SCHEDULER_OP_RUNQUEUE_SIZE = 7`, request/reply structs
- `framework/scheduler_dispatch.h` — dispatch case that calls `ops->runqueue_size(self)`

### Scheduler implementations

**`sched_rr_runqueue_size()`** — walks the single runqueue, counts every task that is not `g_idle_task` (kernel build).  In host-test builds (no idle task) every enqueued task is counted.

**`sched_priority_runqueue_size()`** — same, summed across all 8 priority buckets.

Both registered in their respective `nx_scheduler_ops` tables.

### Core driver

`sched_runqueue_size()` added to `core/sched/sched.c` / `sched.h`: thin wrapper through `g_sched_ops->runqueue_size()`; returns 0 if the scheduler is not yet initialised or the op is NULL.

### Idle loop (`core/boot/boot.c`)

```c
for (;;) {
    asm volatile("wfi");
    if (sched_runqueue_size() > 0)
        nx_task_yield();
}
```

After `wfi` wakes (on any IRQ), the idle task checks whether non-idle tasks are runnable.  If so, it yields immediately rather than looping back into `wfi`.  The existing `need_resched` + IRQ-return mechanism already handles most cases; this adds a belt-and-suspenders check that covers the window between the IRQ return path completing and the idle loop's next `wfi`.

## Key Findings

- The IPC message type generated by the IDL for `runqueue_size` is currently unused: like all other scheduler ops (`tick`, `pick_next`, `yield`), the core driver calls `g_sched_ops->runqueue_size()` directly rather than via IPC messages.  The generated dispatch case is latent infrastructure consistent with the existing convention.
- `sched_runqueue_size()` is a pure read (O(n) list walk); it never yields or blocks.  The yield call is separate and explicit in the idle loop.
- The idle task is always enqueued as a permanent fallback, so it is excluded from the count; `> 0` correctly means "there is real non-idle work available."

## Decisions Made

- **IDL route for `runqueue_size`** — consistent with the existing convention that scheduler ops live in the IDL even though the core driver calls them via direct function pointers.  Keeps `verify-iface-fresh` clean and makes the op IPC-callable as a latent capability.
- **Exclude idle from the count in both schedulers** — lets callers use `> 0` semantics without a `> 1` adjustment; the exclusion is guarded by `#if !__STDC_HOSTED__` so host tests count all tasks (idle is never enqueued in host builds).
